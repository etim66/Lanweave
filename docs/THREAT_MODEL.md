# Threat Model

## Scope

The LAN and discovery channel are hostile. An attacker may advertise false peers, observe endpoints, timing, and traffic volume, capture, modify, delay, replay, or drop ciphertext, terminate connections, open many connections, and send arbitrary framed bytes. This model assumes honest, uncompromised endpoints, a working OS CSPRNG and TLS 1.3 implementation, and a user who privately conveys and enters the pairing string on the intended devices. Endpoint compromise is out of scope.

Protocol version 1 selects RFC 9382 SPAKE2-P256-SHA256, sender A and receiver B, with HKDF-SHA256, HMAC-SHA256 mutual confirmation, and TLS-exporter plus `hello` binding as its target profile. The exact `hello` JSON body bytes, excluding frame headers, are AAD rather than part of RFC 9382 transcript `TT`; each body is capped at 4,000 bytes so the constructed AAD remains within the RFC bound. TLS 1.3 uses ALPN `lanweave/1`, an ephemeral per-connection self-signed ECDSA P-256 certificate, no resumption, and no early data. TLS cipher-suite and key-exchange negotiation remains standard but is constrained to one pinned, reviewed provider policy; Lanweave and PAKE profile negotiation is forbidden. This is a fixed experimental selection, not a claim of audit or security. Production assurance remains release-blocked on specialist review of the RFC mapping, TLS/exporter composition, provisional-certificate verifier, and provider policy and on an audited RFC-conformant Rust dependency; no currently evaluated crate is approved.

## Threats and controls

| Threat | Required control | Residual risk |
| --- | --- | --- |
| Spoofed discovery or provisional endpoint | Treat discovery, addresses, ephemeral certificate, and `hello` as hostile; disclose no manifest before pairing; require successful mutual confirmation before `transfer_request` | An attacker can impersonate availability, collect bounded online attempts, or deny service, but a provisional endpoint does not see names or sizes |
| Split TLS MITM or exporter mismatch | Put the live exporter and both exact `hello` JSON body byte strings, without frame headers, in bounded RFC 9382 AAD rather than `TT`; fail closed on confirmation mismatch | Mapping and composition require specialist review; implementation defects could defeat the binding |
| Pairing-string interception or disclosure | Receiver generates a uniform eight-digit string and shares it over a private out-of-band path; never transmit it as a plain field, persist it, or log it | Anyone who obtains the live string during its validity can defeat pairing; physical observation and compromised OOB channels remain risks |
| Online guesses and reconnect race | Expire after 120 seconds; one use; count at most five failed sender-confirmation verifications across reconnects; atomically reserve/count confirmations and consume or invalidate state; separately bound connections, TLS handshakes, PAKE work, and malformed-input work | The small password space provides no protection after disclosure; attackers can exhaust the ceremony or consume separately bounded work and cause denial of service |
| Pairing-record replay | Bind confirmation to fresh per-connection TLS exporter, AAD, and exact transcript; reject duplicate or wrong-state records; best-effort zeroize connection-local secret state on termination without discarding a still-usable ceremony | Captured ciphertext and records can still be replayed to consume bounded parsing and connection resources; complete erasure cannot be guaranteed |
| Provisional certificate misuse | Generate a fresh self-signed ECDSA P-256 certificate/key for each connection; relax only trust-anchor/name checks; strictly validate certificate structure and signatures, including TLS `CertificateVerify`; do not treat it as identity, reuse it, resume its sessions, or accept early data | The custom verifier and pinned TLS provider policy are release-review gates; generation and connection floods consume bounded resources |
| Malformed PAKE points or encodings | Enforce exact lengths and canonical encodings; reject invalid, identity, non-curve, or non-group elements before use; rely only on an approved constant-time RFC-conformant implementation | No currently evaluated Rust crate is approved; parser, validation, or side-channel bugs may remain |
| Cryptographic error oracle | Use `invalid_message` for malformed messages, encodings, or points; use only coarse `authentication_failed` for valid correctly ordered sender-confirmation failures and, when safe, unavailable ceremonies, which may fail before the Party B share | The common code does not equalize phase, timing, or resource use and does not hide whether a ceremony exists |
| Secret or transcript logging | Exclude pairing strings, PAKE values and keys, exporter values, confirmation tags, ephemeral TLS keys, and full transcripts from logs and crash reports; perform best-effort in-process zeroization on terminal outcomes | Endpoint compromise, allocator/compiler/OS copies, memory disclosure, or unsafe diagnostics can still reveal secrets and are out of scope |
| Unseen transfer or confused consent | Pair first, then send the manifest over the authenticated TLS channel; require separate explicit receiver approval and destination preparation before an accepting `transfer_response`, then send `ready` | Pairing authenticates a channel, not the later transfer; a user can still approve malicious or unexpected content |
| Passive observation and metadata privacy | Start provisional TLS before `hello`; reveal no manifest before pairing; minimize discovery and control metadata | Other LAN devices still observe discovery, endpoints, connection timing and volume, and can capture/drop ciphertext or cause denial of service |
| Downgrade or profile substitution | Require exact protocol/frame version 1, exact ALPN, and one fixed Lanweave/PAKE profile with no capability or profile negotiation; allow only standard TLS 1.3 cipher/KX negotiation inside the pinned reviewed provider policy | Provider drift, implementation bugs, or future unsafe fallback can reintroduce downgrade |
| Connection, prompt, CPU, memory, or disk denial of service | Global/per-source connection and pending-request limits, timeouts, cheap checks first, bounded queues/manifests/storage, rate-limited UI | Coordinated peers can consume LAN bandwidth and some bounded work; availability is not guaranteed |
| Malicious frame lengths or slow frames | Validate the complete 12-byte header and fixed frame-kind maximum before allocation/parsing; checked arithmetic and staged read deadlines | Parser/library bugs may remain |
| JSON duplicate, depth, number, escape, or UI abuse | UTF-8 only; reject duplicate/unknown fields, floats, out-of-range integers, excess fixed wire depth/count/length, malformed escapes, and trailing data; parse every control within fixed maxima before applying lower local manifest/storage admission policy; escape hostile display/log text | Unicode confusables and implementation-specific decoder flaws remain |
| Wrong-state or excess DATA | Accept raw DATA only for the expected current file and within remaining declared size; fail on premature/late DATA or `file_end` | A malicious sender can send harmful but correctly framed file content |
| Destination traversal, aliases, and collisions | Single safe components under a local root; reject unsafe, duplicate, platform-equivalent, reserved, or existing names; create-new/no-follow temp files and atomic finalization | Platform filesystem equivalence and local race behavior are difficult to model perfectly |
| Source changes during send | Check regular-file identity, type, and size before transfer; use available platform change detection and abort on detected inconsistency | Filesystems without snapshots cannot guarantee a stable source against a hostile local process |
| Transmission or storage corruption | TLS integrity plus exact size and digest verification before finalization | A dishonest sender can still choose harmful or unexpected same-sized content; digests do not make content safe |
| Transport interruption, cancellation, or write failure | Stop at first failure, delete the current temp, retain only the verified finalized prefix, and never infer success from EOF | Multi-file transfer is not atomic; retained prefix reveals partial completion and consumes storage |
| Metadata and log disclosure | Never advertise code or ceremony state; send the manifest only after pairing; redact secrets, paths, stable IDs, content, and hostile strings | The paired endpoint learns the offered manifest; LAN observers still learn discovery, endpoints, timing, and volume |

## Residual risks

Lanweave cannot protect a compromised endpoint, prevent a user from opening a malicious received file, prove a person's real-world identity, protect a disclosed pairing string, or guarantee availability on a jammed or filtered network. TLS before pairing does not identify the peer. Pairing authenticates the live channel but does not approve an unseen transfer, so the manifest requires separate receiver consent. Verified files may still be malicious, and a completed prefix remains after a later file fails.

QUIC, custom application AEAD, algorithm agility, persistent trusted devices, resume, parallel transfer, directories, compression, and automatic acceptance require a new threat review before inclusion.
